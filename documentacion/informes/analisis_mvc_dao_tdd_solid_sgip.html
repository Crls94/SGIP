Analisis MVC, DAO, TDD y SOLID
Proyecto SGIP - Metro Ica

Proyecto: SGIP - Sistema de Gestion Inteligente de Inventarios y Pedidos.

Base verificada: backend Spring Boot ubicado en sgip-backend.

Tecnologias identificadas: Java 21, Spring Boot, Spring Web MVC, Spring Data JPA, PostgreSQL, JUnit.

Parte 1: Analisis

1. Elementos del patron MVC

En SGIP Metro, el sistema se organiza usando una arquitectura web donde el frontend React actua como Vista, los controladores REST de Spring Boot actuan como Controlador, y las entidades, DTOs y servicios representan el Modelo y la logica del negocio.

Capa MVC Elementos en SGIP Metro Evidencia en el proyecto
Modelo (Model) Representa los datos y reglas principales del negocio: productos, pedidos, movimientos de inventario, usuarios, alertas, notificaciones y predicciones. Producto.java, Categoria.java, Proveedor.java, Pedido.java, PedidoDetalle.java, InventarioMovimiento.java, Usuario.java, AlertaStock.java, Notificacion.java, PrediccionDemanda.java
Vista (View) Es la interfaz que usa el usuario para interactuar con el sistema. En SGIP corresponde al frontend React. Pantallas documentadas del sistema: Login, Dashboard, Productos, Proveedores, Movimientos, Pedidos, Alertas, Notificaciones, Inteligencia y Usuarios.
Controlador (Controller) Recibe las solicitudes HTTP desde la Vista, valida la entrada y delega la logica al servicio correspondiente. ProductoController.java, PedidoController.java, MovimientoController.java, AuthController.java, UsuarioController.java, AlertaStockController.java, DashboardController.java, NotificacionController.java, InteligenciaController.java

Ejemplo concreto del flujo MVC en productos:

Vista React: Pantalla de Productos
        -> Controller: ProductoController
        -> Service: ProductoService
        -> DAO/Repository: ProductoRepository
        -> Base de datos: PostgreSQL

2. Datos a almacenar y acceso usando DAO

SGIP necesita almacenar datos operativos y administrativos para controlar inventario, pedidos, usuarios y alertas. El acceso a estos datos se realiza mediante el patron DAO. En este proyecto, el DAO esta implementado con Spring Data JPA usando interfaces Repository que extienden JpaRepository.

Dato a almacenar Clase modelo DAO / Repository
Productos del inventario Producto ProductoRepository extends JpaRepository<Producto, UUID>
Categorias de productos Categoria CategoriaRepository extends JpaRepository<Categoria, Integer>
Proveedores Proveedor ProveedorRepository extends JpaRepository<Proveedor, Integer>
Movimientos de inventario InventarioMovimiento MovimientoRepository extends JpaRepository<InventarioMovimiento, UUID>
Pedidos Pedido PedidoRepository extends JpaRepository<Pedido, UUID>
Detalle de pedidos PedidoDetalle PedidoDetalleRepository extends JpaRepository<PedidoDetalle, UUID>
Usuarios y sesiones Usuario, Sesion UsuarioRepository, SesionRepository
Alertas de stock AlertaStock AlertaStockRepository
Notificaciones Notificacion NotificacionRepository
Predicciones de demanda PrediccionDemanda PrediccionRepository

Ejemplo de DAO real en el proyecto:

@Repository
public interface ProductoRepository extends JpaRepository<Producto, UUID> {
    Optional<Producto> findBySku(String sku);

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Producto p WHERE p.id = :id")
    Optional<Producto> findByIdWithLock(@Param("id") UUID id);
}

Con este DAO, la aplicacion puede realizar operaciones CRUD:

productoRepository.save(producto);      // Crear o actualizar
productoRepository.findById(id);        // Buscar por ID
productoRepository.findAll(pageable);   // Listar
productoRepository.delete(producto);    // Eliminar

3. Prueba TDD propuesta antes de codificar

Una prueba TDD importante para SGIP debe enfocarse en una regla critica del negocio: no permitir salidas de inventario cuando no hay stock suficiente.

Prueba elegida:

Verificar que no se pueda registrar una salida de inventario si la cantidad solicitada supera el stock disponible.

Escenario:

Dado un producto con stock actual de 5 unidades,
cuando un usuario intenta registrar una salida de 10 unidades,
entonces el sistema debe rechazar la operacion y mostrar el mensaje "Stock insuficiente".

Relacion con el codigo real: esta regla pertenece a MovimientoService.registrarMovimientoReal(), donde se valida que una salida o merma no supere el stock disponible.

Evidencia de pruebas JUnit existentes:

  • src/test/java/com/metroica/sgip_backend/config/JwtUtilTest.java: verifica generacion y validacion de tokens JWT.
  • src/test/java/com/metroica/sgip_backend/seguridad/UsuarioPrincipalTest.java: verifica la creacion correcta del principal de seguridad.
  • src/test/java/com/metroica/sgip_backend/SgipBackendApplicationTests.java: prueba de contexto, deshabilitada porque requiere PostgreSQL en ejecucion.

4. Principio SOLID para evitar que una clase haga demasiadas cosas

El principio SOLID principal para este caso es SRP - Single Responsibility Principle, o Principio de Responsabilidad Unica.

Este principio indica que una clase debe tener una sola razon para cambiar. En SGIP se aplica separando responsabilidades:

Clase Responsabilidad
ProductoController Recibir solicitudes HTTP relacionadas con productos.
ProductoService Aplicar reglas de negocio del modulo de productos.
ProductoRepository Acceder a datos de productos en PostgreSQL.
ProductoMapper Convertir entidades a DTOs y DTOs a entidades.
ProductoCreateDTO y ProductoResponseDTO Transportar datos de entrada y salida sin exponer directamente la entidad.

3. Prueba TDD elegida en una frase

El sistema debe impedir una salida de inventario cuando la cantidad solicitada sea mayor al stock actual del producto.

4. Principios SOLID que se respetaran

Principio SOLID Aplicacion en SGIP
SRP - Responsabilidad Unica Cada clase cumple una funcion especifica: Controller recibe peticiones, Service aplica reglas, Repository accede a datos, Mapper convierte datos y DTO transporta informacion.
DIP - Inversion de Dependencias Los servicios dependen de interfaces Repository y no de consultas SQL directas ni de una implementacion concreta de base de datos.
OCP - Abierto/Cerrado El sistema puede extender nuevos modulos, reportes o reglas de negocio sin modificar completamente las clases existentes.

Conclusion

SGIP Metro aplica una arquitectura MVC apoyada por servicios, DTOs, mappers y DAOs mediante Spring Data JPA. La Vista corresponde al frontend React; los Controladores son clases REST en Spring Boot; el Modelo esta formado por entidades como Producto, Pedido, Usuario y Movimiento; y el acceso a datos se realiza mediante repositories que funcionan como DAO. Para mantener codigo limpio, el diseno respeta principalmente SRP, DIP y OCP. Como prueba TDD prioritaria se propone validar que no se permita una salida de inventario cuando el stock sea insuficiente.
